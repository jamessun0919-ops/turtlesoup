# 議題 A：「是也不是」判定 — 認知落差（時代／物種）
================================================================
================================================================

## 2026-07-08：視角矛盾判定 -> 應回答「是也不是」

情境：
題目「絕望歌聲」（questions.json id 5，楚漢相爭、四面楚歌的故事）。
玩家問「歌是耳熟能詳的歌嗎」，AI 原本回答「是」。

問題：
這首歌（楚地民謠）對故事中的角色（項羽／楚軍將士）而言是耳熟能詳的家鄉歌謠；
但對玩家（現實世界的推理者）而言，這是一首玩家不會知道、不熟悉的古代歌謠。
同一個問題，代入「角色視角」與「玩家視角」兩種立場，答案是互相矛盾的（一個是「是」，一個是「否」）。

開發者提出的規則：
AI 回答玩家提問的答案，如果對玩家的意義跟對角色的意義有矛盾，應回答「是也不是」。

範例：
- 提問：「歌是耳熟能詳的歌嗎？」
- 代入角色視角：是（楚地鄉音，將士們都熟悉）
- 代入玩家視角：否（現實中玩家不會知道這首古代歌謠）
- 判定：是也不是

套用位置：
- rules_host.txt 第 1 節「核心回答規範」的「是也不是」定義，補充視角矛盾判定的觸發條件與範例說明。
- rules_hint.txt 第 1 節「回答規範」同步補充「視角矛盾判定」，並允許在回答「是也不是」時附一句委婉說明是哪一種視角上的落差。

Agent 的建議（尚未採用，待與開發者進一步討論）：

現行規則風險：
1. 範疇過寬 — 海龜湯裡玩家本來就是故事外的人，任何涉及背景設定（時代、文化、專業知識）的提問都天生存在「玩家不知道／角色知道」的落差。若照規則字面執行，AI 可能把大量原本該回答「是」或「否」的提問都判成「是也不是」，稀釋掉是非題該有的明確訊號，這是遊戲機制的核心。
2. 與規則 3（資訊邊界原則）自相矛盾 — 該節明訂 AI 只能依據湯面/湯底內容判定，不可引入故事外資訊；但視角矛盾判定要求 AI 額外評估「玩家本人是否知道」，這是故事文本以外的資訊，等於引入了規則 3 明文禁止的判斷維度。
3. AI 無法真正判斷「玩家會不會知道」— LLM 沒有真實玩家的背景資訊，所謂「對玩家而言熟不熟悉」只是模型自己的推測，同類問題不同次執行可能給出不一致的答案，玩家會覺得判定隨性。

建議方向：
把觸發範圍收窄，不用「玩家視角 vs 角色視角」這麼寬的框架，改成限定在「湯底文字本身內部就對某個性質給出矛盾或雙重解釋」的情況才判「是也不是」。針對「耳熟能詳」這類主觀感受詞，建議回歸規則 3，直接以故事內角色的認知作答（此例應為「是」），因為玩家個人是否認得這首歌，本來就不屬於湯底該對外揭露的資訊維度。

（此建議尚未套用至 rules_host.txt / rules_hint.txt，待與開發者討論後再決定是否修改。）

## 2026-07-08（後續定案）：視角矛盾判定 -> 收斂為「認知落差判定」

開發者裁定：
確認此範例（絕望歌聲）的成因是故事角色與現實玩家存在**明顯歷史年代差距**，才會出現認知落差；
檢視題庫後目前沒有其他類似情況。因此不採用原本「任何玩家/角色視角矛盾」的寬鬆版本，
改為只在「湯底判斷可能出現歷史年代差距、物種差距（其他動物、外星人、靈異鬼怪）」時，才套用此條規則。

Agent 建議並經開發者確認採用的收斂條件（雙重限制，缺一不可）：
1. 情境限制：湯底故事明確設定「時代落差」（角色所處時代與現代明顯不同，含古代與未來/科幻背景兩種方向）
   或「物種／身分落差」（角色並非現代人類，例如動物、外星人、鬼怪等）。
2. 提問類型限制：玩家問的必須是「主觀感受性質」的提問（熟悉、常見、正常、稀奇、驚訝等形容詞性質），
   而非客觀事實提問（是否死亡、是否使用某物、是否發生某事等）。即使情境有時代/物種落差，
   客觀事實提問仍一律依「核心回答規範」正常判定，不套用本條。

為何這樣收斂：
- 情境限制把觸發條件從「幾乎任何提問都可能有的玩家/角色落差」收窄成兩種明確、可由湯底直接檢查的具體設定，
  大幅降低誤觸發機率，避免稀釋是非題的判定訊號。
- 判斷依據仍完全來自湯底文字本身揭露的設定（時代/物種），不是要 AI 揣測「這個玩家本人知不知道」，
  因此不再與規則 3（資訊邊界原則：只能依湯面/湯底內容判定）互相矛盾。
- 提問類型限制排除掉客觀事實類提問，避免 AI 只因為場景有時代/物種落差，就連無關的客觀提問也套用本條。

套用位置：
- rules_host.txt 第 1 節新增獨立小節「認知落差判定」，取代原本寫在「是也不是」定義裡的視角矛盾判定文字。
- rules_hint.txt 第 1 節「特別限制」後方同步新增「認知落差判定」，套用同樣的雙重限制條件。

狀態：已套用至規則檔案，並已用真實 Gemini API 測試驗證（無提示模式，非 mock）：

| 題目 | 提問 | 回覆 | 驗證重點 |
|---|---|---|---|
| 絕望歌聲（有時代落差） | 歌是耳熟能詳的歌嗎（主觀感受） | 是也不是 | 目標情境仍正確觸發認知落差判定 |
| 絕望歌聲（有時代落差） | 虞姬有拔劍自刎嗎（客觀事實） | 是 | 客觀事實提問正確不觸發本規則，未被誤判 |
| 致命新鞋（無時代/物種落差） | 她的工作是很特殊的職業嗎（主觀感受） | 是 | 無落差情境下不會誤觸發 |

三組結果均符合預期，確認雙重限制條件（情境限制 + 提問類型限制）有效防止過度觸發，
同時保留原本要修正的案例，未再發現規則衝突或誤判。

# 議題 B：開放式問題判定 — 誤判合法是非題為開放式問題
================================================================
================================================================

## 2026-07-08：「誰的聲音有影響嗎」被誤判為開放式問題 -> 改用「代換測試法」

情境：
玩家問「誰的聲音有影響嗎」，AI 判定為開放式問題，回覆「遊戲只能詢問是非題喔」拒答。
對應題目推測為「耳邊的女聲」（questions.json id 8）：湯底是藍牙耳機斷線提示音，並非真人說話。
這句提問其實是很關鍵的推理提問（是否為人聲），理論上應可回答「否」，卻被擋下。

原因分析：
現行規則「特別限制」的開放式問題判定，是用關鍵字表面比對（規則裡列了「是誰」作為範例觸發詞）。
AI 很可能只要偵測到句子中出現疑問詞「誰」，就直接歸類為開放式問題，
沒有進一步判斷「誰的」在這句話裡是修飾語（等同於「某人的」），而非提問的核心目的，
句尾的「嗎」其實才是整句的是非題標記。這跟「他是誰」（拿掉「誰」問題就不成立）在語法功能上不同，
不該套用同一條拒答規則。

開發者裁定：
採用 Agent 提出的分析與修改方向，與議題 A（認知落差判定）是完全不同的規則面向，
議題 A 處理「該不該回答是也不是」，議題 B 處理「該不該一開始就拒答」，兩者觸發條件互不相關，
故獨立分類記錄，避免未來誤以為是同一次規則變更的延伸。

Agent 提出並採用的規則修改（代換測試法）：
將句中的疑問詞（誰／什麼／哪裡等）代換成不定代名詞（某人／某個東西／某處），
檢查整句是否仍是一個通順、可用是/否/是也不是/無關回答的完整是非題：
- 代換後仍通順可回答 -> 視為合法是非題，正常回答。
  例：「誰的聲音有影響嗎」-> 代換為「某人的聲音有影響嗎」-> 通順可回答 -> 視為合法是非題。
- 代換後問題就不成立或失去意義 -> 才是真正的開放式問題，回覆「遊戲只能詢問是非題喔」。
  例：「他是誰」-> 代換為「他是某人」-> 失去提問意義 -> 維持拒答。

套用位置：
- rules_host.txt「特別限制」小節，加入代換測試法的判斷步驟與正反範例。
- rules_hint.txt「特別限制」小節同步加入相同的代換測試法。

狀態：已套用至規則檔案，並已用真實 Gemini API 測試驗證（無提示模式，非 mock）：

| 題目 | 提問 | 回覆 | 說明 |
|---|---|---|---|
| 致命新鞋 | 誰的鞋子造成了影響嗎 | 是 | 修正生效，正確識別為合法是非題 |
| 致命新鞋 | 兇手是誰 | 遊戲只能詢問是非題喔 | 真開放式問題仍正確拒答 |
| 耳邊的女聲 | 誰的聲音有影響嗎（原始回報例句） | 遊戲只能詢問是非題喔 | 原始例句仍被拒答，修正未生效 |
| 耳邊的女聲 | 某人的聲音有影響嗎（手動代換後） | 是也不是 | 代換後語句本身沒有問題，AI 能正常回答 |

結果為部分成功、非全面解決：
- 規則對「誰的鞋子造成了影響嗎」有效，但對回報的原始例句「誰的聲音有影響嗎」仍然失效——
  同樣是「誰的 X 有/造成了影響嗎」句型，AI 兩次判斷不一致。
- 手動把「誰」代換成「某人」後（第 4 組），AI 就能正常回答，證明句子本身沒有語意問題，
  而是 AI 在偵測到「誰」這個字時，沒有穩定執行代換測試步驟就先拒答——推測是規則檔案裡
  「特別限制」的拒答範例列在代換測試法之前，AI 傾向優先比對到「是誰」類範例就直接拒答，
  沒有繼續往下執行代換測試的邏輯。
- 這代表目前修正「有改善但不完全可靠」：同句型會因用詞細節（「造成了影響」vs「有影響」）
  在不同次執行judgment不一致，這也是 LLM 作為分類器的固有限制，不是單純把規則文字寫得更完整
  就能徹底消除的問題。

開發者裁定：接受目前改善幅度即可，不再進一步調整規則優先順序或重測。
（可能的後續加強方向：把代換測試法段落移到拒答範例之前，讓 AI 優先執行代換測試——
若未來又發現類似誤判案例，可從這個方向繼續嘗試。）

## 2026-07-08（後續實驗）：調換段落順序 -> 未能穩定改善，已改回原順序

實驗：
把「是非題判斷方法（代換測試法）」段落移到「特別限制」（拒答範例）之前，
讓 AI 理論上會先執行代換測試，再參考拒答範例。

測試結果：

| 順序 | 題目 | 提問 | 回覆 |
|---|---|---|---|
| 調整後，第 1 次 | 耳邊的女聲 | 誰的聲音有影響嗎 | 是（原本失敗的案例修好了） |
| 調整後，第 1 次 | 致命新鞋 | 誰的鞋子造成了影響嗎 | 遊戲只能詢問是非題喔（原本成功的案例壞掉了） |
| 調整後，第 1 次 | 致命新鞋 | 兇手是誰 | 遊戲只能詢問是非題喔（控制組維持正確） |
| 調整後，重複測試 | 致命新鞋 | 誰的鞋子造成了影響嗎 | 遊戲只能詢問是非題喔（與前次一致） |
| 調整後，重複測試 | 耳邊的女聲 | 誰的聲音有影響嗎 | 遊戲只能詢問是非題喔（**與同一輪第 1 次的「是」矛盾**） |

關鍵發現：
同一句「誰的聲音有影響嗎」、同一份規則檔案、同一個 temperature（0.2），前後兩次呼叫給出不同答案（是 -> 拒答）。
這證明**問題不在規則文字寫得夠不夠好、或段落順序對不對，而是這類邊界句型本身，LLM 每次判斷就存在隨機浮動**，
調換順序只是把「哪一個案例會失敗」換了位置，整體沒有淨改善——修好一個案例的同時，弄壞了另一個原本正常的案例。

開發者裁定：
- 段落順序改回原本版本（拒答範例在前，代換測試法在後）。
- 記錄為「尚未完全改善」，不是靠這次調整就解決的問題。
- 現階段 LLM 對玩家的不標準提問（語意表達方式與開發者預期不同）判斷本來就會有隨機浮動，
  需要累積更多實際遊玩中出現的例句，才能進一步歸納出更有效的規則寫法或其他機制。

問題的本質（開發者觀點，記錄供後續研究參考）：
這個問題的根源不是規則寫得不夠精確，而是**玩家（尤其是遊戲情境下的一般使用者）提問時，
語言表達不會像開發者預期的那樣標準化，語意表達的混亂性本來就很高**。單純靠在系統提示詞裡
補規則文字，只能處理「已經想到的」邊界案例，無法涵蓋所有玩家可能問出的不標準句型。
如何建立一套機制，讓 LLM 面對這種高度不規則的自然語言輸入時，仍能穩定判斷是非題的意圖，
是需要持續觀察實際遊玩紀錄、累積更多真實案例後才能研究的待解問題，非本次可以定案。

狀態：本議題（開放式問題判定 / 代換測試法）暫時保留現有規則文字（含代換測試法，順序已改回原狀），
不再進一步調整，等待後續累積更多實際案例。

## 2026-07-08（重要校正 + 新修正措施）：溫度歸零 + 是非題標記改由程式碼判定

**對上一則紀錄的校正**：
上一則「調換段落順序」實驗中，「同一句話、同 temperature、前後兩次答案矛盾」的證據，
事後發現當時 Gemini 呼叫已經觸發免費額度限制（`quota exceeded, limit: 20`），
系統依既有的容錯設計自動降級改用備援 OpenAI（gpt-4o-mini）。
也就是說，前後兩次呼叫**很可能是兩個不同引擎（Gemini vs OpenAI）在回答，不是同一個引擎的隨機浮動**。
這代表上一則紀錄「LLM 對這類邊界句型的判斷存在隨機浮動」的結論混入了引擎切換這個變因，
證據力比原本記錄的弱，特此校正、但不刪除原始紀錄（保留完整討論過程）。

開發者提出的下一步建議（承接前述「使用者語意不標準」問題的討論）：
1. 把 `temperature` 從 0.2 調到 0（Gemini、OpenAI 皆同），降低單一引擎內部的抽樣浮動。
2. 把「是否為合法是非題」的判斷部分移出 LLM 自行判斷，改用程式碼端的決定性規則：
   偵測提問句尾是否有明確是非題標記（嗎／是不是／是否／有沒有／對不對），若有，
   不論句中是否出現「誰」、「什麼」、「哪裡」，一律視為合法是非題；由伺服器動態組裝一段
   「系統額外指令」附加到 systemInstruction 最後，明確告知 AI 本次提問已判定為合法是非題，
   不可拒答。

套用位置：
- server.js 新增 `hasYesNoMarker()` 函式（正則 `/(嗎|是不是|是否|有沒有|對不對)/`）。
- `callGemini` / `callOpenAI` 的 `temperature` 皆由 0.2 改為 0。
- `/api/chat` 處理流程中，若偵測到最新提問有是非題標記，動態在 systemInstruction 尾端
  附加強制覆寫指令；Mock Mode 的開放式問題判定也同步加上「有是非題標記則不算開放式問題」的排除條件。

測試結果（temperature=0，因 Gemini 額度已於本輪測試中打完，最終測試對象實為備援 OpenAI gpt-4o-mini）：

| 提問 | 第 1 次 | 第 2 次（重複） |
|---|---|---|
| 誰的聲音有影響嗎 | 遊戲只能詢問是非題喔 | 遊戲只能詢問是非題喔（一致） |
